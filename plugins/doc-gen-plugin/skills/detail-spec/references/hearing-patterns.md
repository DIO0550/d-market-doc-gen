# ヒアリングパターン - 詳細仕様書

---

# Part 1: 深掘りモード

固定質問ではなく、テンプレートごとの**必須トピック・深掘り指針・不十分検知ルール**を定義する。各トピックについてユーザーの回答に応じて動的に追加質問を生成し、十分な情報が得られるまで掘り下げる。

## 共通ルール

### 質問の原則

- 1回の AskUserQuestion で1〜3問。多すぎると回答が雑になる
- **ユーザーの回答を要約して確認**してから次のトピックに進む（「〜という理解で合っていますか？」）
- **曖昧シグナルを検知**したら必ず掘り下げる（下記参照）
- トピックの順序は柔軟に。ユーザーの話の流れに合わせる
- 全必須トピックについて**不明点が完全にゼロになるまで**ドラフト生成に進まない

### 曖昧シグナルと対応

| シグナル | 対応 |
|:---------|:-----|
| 「たぶん」「〜だと思う」「おそらく」 | 「根拠は何ですか？」「確定していない場合、どちらに倒しますか？」 |
| 「まだ決めてない」「未定」 | 「仮で決めるとしたらどちらですか？」「この仕様書で決めてしまいましょうか？」 |
| 「なんでもいい」「お任せ」 | 選択肢を2-3個提示し「どれが近いですか？」 |
| 回答が1文だけ | 「もう少し詳しく教えてください。具体的には〜」 |
| 代替案なしの判断 | 「他に検討した案はありますか？その案を不採用にした理由は？」 |

### 十分性の判断基準

あるトピックが「十分」と判断するための基準：
- **不明点が完全にゼロ**であること。1つでも「たぶんこうだろう」と推測で埋める箇所があるなら掘り下げが足りない
- 仕様書の対応セクションを、推測なしに具体的に書けるだけの情報がある
- 曖昧なポイント・未確認の前提が残っていない
- ユーザーが意図的に「対象外」「該当なし」と明言している場合のみスキップ可
- **わからないことがわからない状態は最も危険**。トピックごとに「このトピックで自分がまだ知らないことはないか？」とセルフチェックする

## B6: 設計仕様書 — 深掘りトピック

### 必須トピックチェックリスト

- [ ] **背景と課題**: 現状・問題・この設計で解決すること
- [ ] **スコープ**: 対象/対象外/境界条件
- [ ] **用語定義**: コード上の命名に影響する用語の統一
- [ ] **設計判断**: 各判断の内容・理由・代替案・トレードオフ
- [ ] **ドメインモデル**: エンティティ・属性・関係・不変条件
- [ ] **振る舞い**: ユースケース（正常・代替・例外フロー）
- [ ] **状態遷移**: 状態を持つエンティティの遷移ルール
- [ ] **データフロー**: 入力→処理→出力の流れ
- [ ] **エラー設計**: エラーケースと対処方針
- [ ] **非機能要件**: パフォーマンス・セキュリティ・可用性
- [ ] **制約条件**: 技術的/ビジネス的制約
- [ ] **未決事項**: 実装前に解決すべき未決定事項

### 深掘り指針

**背景と課題**:
- 「なぜ今このタイミングで？」と聞いて緊急度・優先度を把握
- 既存のシステムがある場合: 現在のアーキテクチャ、問題の発生頻度や影響度を聞く
- 「この設計がないと何が困るか？」で課題の本質を確認

**設計判断**:
- 最も重要なトピック。各判断について必ず以下を確認:
  - なぜその選択をしたか（理由）
  - 何を検討して捨てたか（代替案）
  - 何を受け入れたか（トレードオフ）
- 判断が1つしかない場合: 「他に重要な設計判断はありませんか？例えばデータモデル、技術選定、コンポーネント分割など」
- 代替案がない場合: 「他のアプローチも考えましたか？例えば〜」と具体例を提示

**ドメインモデル**:
- エンティティを聞いたら: 各エンティティの属性、型、必須/任意を掘り下げる
- 「このエンティティが常に満たすべきルール（不変条件）はありますか？」
- 「エンティティ間の関係の多重度は？1:1? 1:N? N:M?」
- 「カスケード削除は必要ですか？」

**振る舞い**:
- 正常フローを聞いたら: 「正常フローから外れるケースは？」（代替フロー）
- 「このステップが失敗したらどうなりますか？」（例外フロー）
- 「並行実行される可能性は？その場合の競合は？」

**スコープ**:
- 「対象範囲を具体的に列挙してください」
- 「明確に対象外にするものは？」→ 境界が曖昧なら具体例で確認
- 「将来的にスコープに入れる可能性のあるものは？」

**用語定義**:
- 「チーム内で認識がずれやすい用語はありますか？」
- 「コード上の命名とビジネス用語が異なるものは？」
- エンティティ名・イベント名を聞いたら「コード上はどう命名しますか？」

**状態遷移**:
- 「状態を持つエンティティはありますか？」→ あれば全状態を列挙させる
- 各状態について: 遷移条件（誰が/何がトリガー）、遷移先、ガード条件
- 「不正な遷移（例: 完了→下書き）を防ぐルールは？」
- **ステートマシン図（Mermaid stateDiagram-v2）をドラフトに含める**

**データフロー**:
- 「データの入口はどこですか？（UI/API/バッチ/外部連携）」
- 「データの変換・加工ポイントは？」
- 「最終的なデータの出口は？（DB/外部API/画面表示）」

**非機能要件**:
- 「レスポンスタイムの目標は？」「同時アクセス数の想定は？」
- 「セキュリティ上、特に守るべきポイントは？（認証・認可・暗号化等）」
- 「可用性の要件は？（SLA目標・ダウンタイム許容等）」

**制約条件**:
- 「使えない技術・ライブラリはありますか？」
- 「期限・リソースの制約は？」
- 「既存システムとの互換性制約は？」

**未決事項**:
- 「まだ決まっていないが、実装前に決める必要があるものは？」
- 「誰に確認を取る必要がありますか？」

**エラー設計**:
- エラーが2個以下の場合: 「他にエラーになるケースは？ネットワーク障害、データ不整合、権限不足、タイムアウトなどは？」
- 各エラーについて: 深刻度、リトライ可否、ユーザーへの影響を確認

## C5: 実装仕様書 — 深掘りトピック

### 必須トピックチェックリスト

- [ ] **設計コンテキスト**: 何を作るか・なぜこう作るか・やらないこと
- [ ] **技術スタック**: 言語・FW・DB・ビルド/テスト/リントコマンド
- [ ] **変更禁止事項**: 触ってはいけないファイル・パターン
- [ ] **ファイル構成**: 作成/変更するファイル一覧と責務
- [ ] **インターフェース**: 型定義・関数シグネチャ・APIエンドポイント
- [ ] **データ構造**: DBスキーマ・内部データ構造
- [ ] **処理フロー**: 各処理のステップ詳細
- [ ] **エラーハンドリング**: エラーごとの検知・処理・ログ・メッセージ
- [ ] **実装パターン**: 採用パターン・命名規約・コード例
- [ ] **テスト要件**: テストケース・テストデータ・実行方法
- [ ] **受入基準**: GIVEN/WHEN/THEN 形式
- [ ] **エッジケース**: 境界値・異常系
- [ ] **パフォーマンス要件**: 指標・目標値・計測方法
- [ ] **状態遷移**: 状態を持つエンティティのステートマシン定義
- [ ] **完了条件（DoD）**: 自動検証・仕様準拠・制約遵守

### 深掘り指針

**設計コンテキスト**:
- 「設計仕様書はありますか？パスは？」→ あれば Read して確認、なければ設計判断を直接ヒアリング
- 「完成状態を1-3文で説明すると？」→ 曖昧なら「具体的にユーザーが何をできるようになりますか？」
- 「やらないことは？」→ 未定義なら「〜は今回のスコープに含みますか？」と具体例で確認

**技術スタック**（コードベースが手元にある場合）:
- 「package.json/go.mod 等を確認してよいですか？」→ 読んで自動収集
- ユーザーに聞くのは技術的判断が必要な部分のみ

**ファイル構成**:
- 「既存のディレクトリ構成に合わせたいですか？」
- 「似た機能の実装はどこにありますか？参考にしたいです」
- 各ファイルの責務を1文で確認

**インターフェース**:
- 型を聞いたら: 各フィールドの型、null許容、バリデーションルールを掘り下げる
- APIの場合: リクエスト/レスポンスの具体例、エラーレスポンスの形式
- 「既存のインターフェースとの整合性は大丈夫ですか？」

**処理フロー**:
- 各処理について: 入力→バリデーション→処理→保存→レスポンスの各ステップ
- 「このステップでDBトランザクションは必要ですか？」
- 「並行実行時の排他制御は必要ですか？」

**テスト要件**:
- 「正常系と異常系のテストケースを挙げてください」
- 「モック/スタブが必要な外部依存は？」
- 「カバレッジ目標はありますか？」

**変更禁止事項**:
- 「変更してはいけないファイルやディレクトリは？」
- 「既存のアーキテクチャパターンで壊してはいけないものは？」
- 「命名規則や公開APIの後方互換性は？」

**データ構造**:
- 「新規テーブル/コレクションは必要ですか？」
- 「既存テーブルへのカラム追加は？マイグレーション方法は？」
- 「メモリ上のキャッシュやステート構造は？」

**エラーハンドリング**:
- 「各エラーの検知方法は？（try-catch/条件分岐/ミドルウェア）」
- 「リトライすべきエラーとすべきでないエラーは？」
- 「エラー時のログレベルとログに含める情報は？」
- 「ユーザーに見せるエラーメッセージの方針は？」

**実装パターン**:
- 「採用するデザインパターンは？（Repository/Factory/Strategy等）」
- 「既存コードベースのパターンに合わせますか？それとも新しいパターンを導入？」
- 「命名規約は既存に合わせる？具体的な規約は？」

**受入基準**:
- 「GIVEN/WHEN/THEN形式で受入基準を挙げてください」
- 「正常系以外のシナリオは？」
- 基準が少ない場合: 「〜のケースは受入基準に含めますか？」と具体例提示

**エッジケース**:
- 「境界値（0件/1件/上限）での動作は？」
- 「空文字/null/未定義が入力された場合は？」
- 「同時実行・競合状態のケースは？」

**パフォーマンス要件**:
- 「レスポンスタイムの目標値は？」
- 「データ量の想定は？（現在/1年後/最大）」
- 「計測方法は？（ロードテスト/APM/ベンチマーク）」

**状態遷移**:
- 「状態を持つエンティティはありますか？」
- 各状態の遷移条件・ガード条件を確認
- **ステートマシン図（Mermaid stateDiagram-v2）を仕様書に含める**

**完了条件（DoD）**:
- ビルド/テスト/リント/型チェックの具体的コマンドを確認
- 「他に実行すべき検証はありますか？（セキュリティスキャン、E2Eテスト等）」
- 「プロジェクト固有のルール（CIで必須のチェック等）はありますか？」

## C6: 実装前コードベース探索 — 深掘りトピック

### 必須トピックチェックリスト

- [ ] **探索の目的**: 何を実装したいか・何を明らかにしたいか
- [ ] **探索範囲**: プロジェクトルートパス・重点調査領域
- [ ] **関連情報**: 関連モジュール・変更禁止領域・参照ドキュメント

### 深掘り指針

**探索の目的**:
- 「impl-detail-spec を書くためにどんな情報が足りていませんか？」
- 「既存コードのどの部分が不安ですか？」

**探索範囲**:
- 「特に気になるディレクトリやファイルはありますか？」
- 「類似の機能は既にありますか？あればそのパスは？」

# Part 2: テンプレートモード（固定バッチ質問）

素早くテンプレートを埋めたい場合に使用する。バッチ形式で1〜4問ずつ質問する。

---

## B6: 設計仕様書

### バッチ 1: 基本情報
- 設計対象の機能/システム名は？
- 現状どうなっていて、何が課題か？
- この設計で最終的にどうなるべきか？
- スコープ（対象/対象外）は？

### バッチ 2: 設計判断
- 主要な設計判断は？（アーキテクチャ、技術選定、データモデル等）
- 各判断で検討した代替案とその不採用理由は？
- 受け入れたトレードオフは？

### バッチ 3: ドメインモデルと振る舞い
- 主要なエンティティとその関係は？
- エンティティの不変条件（常に満たすべきルール）は？
- 主要なユースケースの正常フロー・代替フロー・例外フローは？
- 状態遷移を持つエンティティはあるか？その遷移ルールは？

### バッチ 4: エラー・制約・非機能
- 起こりうるエラーケースとその対処方針は？
- 技術的/ビジネス的な制約条件は？
- パフォーマンス・セキュリティ・可用性の要件は？
- 未決事項（実装前に解決すべきもの）はあるか？

---

## C5: 実装仕様書

### バッチ 1: 設計コンテキスト
- 実装する機能/プロジェクト名は？
- 完成状態を1〜3文で説明すると？
- 参照する設計仕様書はある？（パスがあれば）
- 主要な設計判断とその理由は？（設計仕様書がない場合）
- やらないこと（スコープ外）は？

### バッチ 2: 技術スタックとファイル構成
- 技術スタック・フレームワーク・バージョンは？
- ビルド/テスト/リントのコマンドは？
- 作成・変更するファイルの一覧と各ファイルの責務は？
- 変更禁止の既存コード/アーキテクチャは？

### バッチ 3: インターフェースと処理フロー
- 公開するインターフェース（型、関数シグネチャ、APIエンドポイント）は？
- 各処理のステップ詳細は？（入力→バリデーション→処理→出力）
- データベーススキーマの変更はあるか？

### バッチ 4: エラー・テスト・受入基準
- 各エラーの具体的な処理方法は？
- 受入基準は？（GIVEN/WHEN/THEN形式で）
- エッジケースは？
- パフォーマンス要件は？

### バッチ 5: 完了条件（DoD）
- 型チェックのコマンドは？（例: `tsc --noEmit`）
- ビルド/テスト/リント以外に実行すべき検証コマンドはあるか？
- プロジェクト固有の完了条件はあるか？（例: カバレッジ閾値、セキュリティスキャン等）

---

## C6: 実装前コードベース探索

### バッチ 1: 探索対象
- 実装したい機能・変更の概要は？
- 対象プロジェクトのルートパスは？
- 特に重点的に調べたい観点は？（既存パターン/データ構造/影響範囲/テスト構造等）

### バッチ 2: 探索の方向づけ
- 関連しそうな既存機能やモジュールは？（分かる範囲で）
- 変更禁止の領域は把握しているか？
- 参照すべき設計ドキュメントはあるか？（パスがあれば）
