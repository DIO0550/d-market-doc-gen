# 詳細仕様書生成

設計仕様書・実装仕様書・実装前探索ドキュメントを対話的に生成するスキル。

**設計方針**: 全テンプレートは「AIエージェントが読んで実装する」ことに特化。仕様書はテンプレートの空欄埋めではなく、**エンジニアの設計判断をAIに伝えるためのレビュー表面**。ユーザーの判断を徹底的に引き出し、AIが解釈の余地なく実装できる精度の仕様書を目指す。

## テンプレート一覧

| ID | テンプレート | パス | 用途 |
|:---|:------------|:-----|:-----|
| B6 | 設計仕様書 | `assets/templates/design-spec.md` | 設計判断・境界・振る舞い・エッジケースを網羅。AIが解釈の余地なく実装できる精度の設計 |
| C5 | 実装仕様書 | `assets/templates/impl-detail-spec.md` | 設計意図+コードレベル実装詳細。ファイル構成・関数シグネチャ・処理フロー・エラー処理まで網羅 |
| C6 | 実装前コードベース探索 | `assets/templates/impl-exploration.md` | impl-detail-spec の前段階。コードベースを探索し技術的事実を収集・記録する |

## 2つのモード

| モード | 特徴 | 適するケース |
|:-------|:-----|:-----------|
| **深掘りモード** | 適応的質問 → ドラフト生成 → レビュー＆修正サイクル | 設計をしっかり考えたい、判断が多い、複雑な機能 |
| **テンプレートモード** | 固定バッチ質問 → テンプレート埋め | 素早く作りたい、既に頭の中が整理されている |

## Step 0: モード選択

### 自動判定キーワード

| キーワード | モード |
|:----------|:------|
| 「じっくり」「深く考えたい」「しっかり」「ちゃんと詰めたい」「練り上げたい」 | 深掘りモード |
| 「サクッと」「テンプレートで」「素早く」「さっと」「雛形で」 | テンプレートモード |

### 判定不能な場合

AskUserQuestion でモードを選択させる。

```
質問: どちらのモードで仕様書を作りますか？
  - 深掘りモード（じっくり対話して仕様を練り上げる）
  - テンプレートモード（質問に答えてテンプレートを素早く埋める）
```

## Step 1: テンプレート選択

### 自動判定キーワード

| キーワード | テンプレート |
|:----------|:-----------|
| 「設計仕様」「設計を詰める」「設計書」「仕様を詰める」「判断根拠」 | B6: 設計仕様書 |
| 「実装仕様」「実装詳細」「コードレベル仕様」「実装設計」 | C5: 実装仕様書 |
| 「実装前探索」「コードベース探索」「実装探索」「コード調査」「事前調査」 | C6: 実装前コードベース探索 |

### 判定不能な場合

AskUserQuestion でテンプレートを選択させる。

```
質問: どのテンプレートを使いますか？
  - 設計仕様書（設計判断・ドメインモデル・振る舞い仕様）
```

# 深掘りモード

## Step 2a: コンテキスト収集

最低限の初期情報を集める。AskUserQuestion で1〜3問。

**B6（設計仕様書）の場合**:
- 何の設計仕様書を作りますか？（機能/システム名）
- 現状どうなっていて、何が課題ですか？

**C5（実装仕様書）の場合**:
- 何を実装しますか？（機能名、完成状態を1-3文で）
- 参照する設計仕様書はありますか？（あれば Read で読む）

**C6（実装前探索）の場合**:
- 何を実装する予定ですか？（概要）
- 対象プロジェクトのルートパスは？

## Step 3a: 適応的深掘りプロービング

`references/hearing-patterns.md` の **Part 1: 深掘りモード** を Read ツールで読み、テンプレートIDに対応する必須トピックと深掘り指針に従う。

### ルール

1. **必須トピックを順に掘り下げる**。ただし順序はユーザーの話の流れに合わせて柔軟に変更してよい
2. **1回の AskUserQuestion で1〜3問**。ユーザーが多くの情報を一度に出してくれた場合は、確認質問だけでよい
3. **回答を要約して確認**してから次のトピックに進む
4. **曖昧シグナルを検知**したら必ず掘り下げる（hearing-patterns.md の「曖昧シグナルと対応」参照）
5. **十分性を判断**する: そのトピックについて仕様書の対応セクションを書けるだけの情報があるか？
6. **全必須トピックが網羅されるまで**ドラフト生成に進まない
7. **ユーザーが「もう十分」と言った場合**はそこで止める。ただし必須トピックに大きな抜けがあれば「〜についてはまだ聞けていませんが、スキップしてよいですか？」と確認

### C5 固有: コードベース調査の統合

実装仕様書の場合、ヒアリングの中で以下を積極的に行う:

- ユーザーが「対象プロジェクト」を示したら、Glob/Grep/Read で既存コードを調査
- package.json, tsconfig.json 等から技術スタックを自動収集
- 類似機能の実装パターンを読み取り、ユーザーに確認（「既存の〜と同じパターンでよいですか？」）
- 自動収集した情報はユーザーに報告し、正しいか確認を取る

## Step 4a: ドラフト生成

1. テンプレートを Read ツールで読む（構成の参考として）
2. ヒアリング結果をもとにドラフトを生成する
3. テンプレートの構成に縛られすぎない。**不要なセクションは削除、必要なセクションは追加**してよい
4. **コードスニペットを可能な限り含む**: 型定義、関数シグネチャ、コード例
5. **具体的なファイルパスを含む**: 作成/変更するファイルの実際のパス
6. Mermaid図がある場合、内容に合わせて正確に描画する
7. **Write ツールでファイルに書き出す**（この時点ではドラフトとして）

## Step 5a: レビュー＆アノテーションサイクル

ドラフトを書き出した後、ユーザーにレビューを依頼する。

```
ドラフトを {ファイルパス} に書き出しました。
確認して、修正したい点があれば教えてください。
- 「〜のセクションを削除して」
- 「〜の判断理由を変えて。理由は〜」
- 「〜のエラーケースが足りない。〜を追加して」
- 「OK」（承認）
のように指示してください。
```

### ルール

- ユーザーの修正指示を受けたら Edit ツールで該当箇所を修正
- 修正後、修正箇所を要約して報告
- ユーザーが「OK」「これでいい」「承認」等と言うまで繰り返す
- 大きな変更の場合は修正後に再度レビューを依頼

## Step 6: 完了報告

最終版のファイルパスと概要を表示する。

# テンプレートモード

## Step 2b: 固定バッチヒアリング

`references/hearing-patterns.md` の **Part 2: テンプレートモード** を Read ツールで読み、テンプレートIDに対応するバッチ質問に従う。

ルール:
- 1回のバッチで1〜4問を AskUserQuestion で質問
- ユーザーが既に回答済みの質問はスキップ
- ユーザーの言語に合わせる

## Step 3b: ドキュメント生成

1. テンプレートIDに対応するファイルを Read ツールで読む
2. ヒアリング結果でテンプレートのプレースホルダーを埋める
3. **不要なセクションは削除**、**必要なセクションは自由に追加**してよい
4. Mermaid図がある場合、内容に合わせて正確に描画する

## Step 4b: ファイル書き出し

**Write ツールで書き出す。**

## Step 5b: 完了報告

書き出したファイルのパスと概要を表示する。
