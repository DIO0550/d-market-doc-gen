# ヒアリングパターン - API仕様書

## 共通ルール

### 質問の原則

- 1回の AskUserQuestion で1〜3問。多すぎると回答が雑になる
- **ユーザーの回答を要約して確認**してから次のトピックに進む（「〜という理解で合っていますか？」）
- **曖昧シグナルを検知**したら必ず掘り下げる（下記参照）
- トピックの順序は柔軟に。ユーザーの話の流れに合わせる
- 全バッチの質問について**不明点が完全にゼロになるまで**ドキュメント生成に進まない
- バッチ質問は「最低限聞くべきこと」。回答から新たな疑問が生まれたら追加質問する

### 曖昧シグナルと対応

| シグナル | 対応 |
|:---------|:-----|
| 「たぶん」「〜だと思う」「おそらく」 | 「根拠は何ですか？」「確定していない場合、どちらに倒しますか？」 |
| 「まだ決めてない」「未定」 | 「仮で決めるとしたらどちらですか？」「この仕様書で決めてしまいましょうか？」 |
| 「なんでもいい」「お任せ」 | 選択肢を2-3個提示し「どれが近いですか？」 |
| 回答が1文だけ | 「もう少し詳しく教えてください。具体的には〜」 |
| 代替案なしの判断 | 「他に検討した案はありますか？その案を不採用にした理由は？」 |

### 十分性の判断基準

あるトピックが「十分」と判断するための基準：
- **不明点が完全にゼロ**であること。1つでも「たぶんこうだろう」と推測で埋める箇所があるなら掘り下げが足りない
- 仕様書の対応セクションを、推測なしに具体的に書けるだけの情報がある
- 曖昧なポイント・未確認の前提が残っていない
- ユーザーが意図的に「対象外」「該当なし」と明言している場合のみスキップ可
- **わからないことがわからない状態は最も危険**。トピックごとに「このトピックで自分がまだ知らないことはないか？」とセルフチェックする

## 共通（全フォーマット）

**バッチ 1: 基本情報**
- API名 / サービス名は？
- このAPIの目的は？
- 利用者は？（フロントエンド、モバイル、外部、他サービス）
- ベースURLまたはサービス境界は？

**バッチ 2: 認証と共通仕様**
- 認証方式は？（Bearer、APIキー、OAuth 2.0、なし）
- レスポンスフォーマットは？（JSON、XML）
- レートリミットの要件は？
- エラーハンドリングのパターンは？

## Markdown API のヒアリング

**バッチ 3: エンドポイント**
- このAPIが管理するリソース / エンティティは？
- リソースごとに必要な操作は？（一覧、取得、作成、更新、削除、カスタム）
- 一覧操作のクエリパラメータは？（フィルタリング、ソート、ページネーション）
- Webhookや非同期操作はある？

## OpenAPI のヒアリング

**バッチ 3: スキーマ詳細**
- 主要なリソーススキーマは？（フィールド、型、必須/任意）
- 操作ごとのリクエストボディスキーマは？
- ステータスコードごとのレスポンススキーマは？
- 再利用可能なコンポーネントはある？（共通ヘッダー、エラースキーマ、ページネーション）

**バッチ 4: 操作**
- 正確なエンドポイントパスとHTTPメソッドは？
- 各エンドポイントのパス/クエリパラメータは？
- 操作ごとのセキュリティスコープは？
- APIバージョニング戦略は？（URLパス、ヘッダー、クエリ）

## GraphQL のヒアリング

**バッチ 3: 型システム**
- 主要な型は？（フィールド、リレーション、NULL許可）
- Enum型は？
- ミューテーション用のInput型は？
- インターフェースやユニオン型はある？

**バッチ 4: 操作**
- 必要なクエリは？（引数、戻り値の型）
- 必要なミューテーションは？（入力、戻り値の型）
- サブスクリプションはある？（イベント、ペイロード）
- カスタムスカラーは？（Date、JSONなど）

## セクション適応ルール

- 認証なし → 認証セクションをスキップ
- ページネーションなし → ページネーションセクションをスキップ
- レートリミットなし → レートリミットセクションをスキップ
- Webhookなし → Webhookセクションをスキップ
- GraphQL: サブスクリプションなし → サブスクリプションセクションをスキップ
- OpenAPI: 単純なCRUD → リソーススキーマから自動生成
