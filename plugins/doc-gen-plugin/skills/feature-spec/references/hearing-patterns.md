# ヒアリングパターン - 機能仕様書 / PRD

## 共通ルール

### 質問の原則

- 1回の AskUserQuestion で1〜3問。多すぎると回答が雑になる
- **ユーザーの回答を要約して確認**してから次のトピックに進む（「〜という理解で合っていますか？」）
- **曖昧シグナルを検知**したら必ず掘り下げる（下記参照）
- トピックの順序は柔軟に。ユーザーの話の流れに合わせる
- 全バッチの質問について**不明点が完全にゼロになるまで**ドキュメント生成に進まない
- バッチ質問は「最低限聞くべきこと」。回答から新たな疑問が生まれたら追加質問する

### 曖昧シグナルと対応

| シグナル | 対応 |
|:---------|:-----|
| 「たぶん」「〜だと思う」「おそらく」 | 「根拠は何ですか？」「確定していない場合、どちらに倒しますか？」 |
| 「まだ決めてない」「未定」 | 「仮で決めるとしたらどちらですか？」「この仕様書で決めてしまいましょうか？」 |
| 「なんでもいい」「お任せ」 | 選択肢を2-3個提示し「どれが近いですか？」 |
| 回答が1文だけ | 「もう少し詳しく教えてください。具体的には〜」 |
| 代替案なしの判断 | 「他に検討した案はありますか？その案を不採用にした理由は？」 |

### 十分性の判断基準

あるトピックが「十分」と判断するための基準：
- **不明点が完全にゼロ**であること。1つでも「たぶんこうだろう」と推測で埋める箇所があるなら掘り下げが足りない
- 仕様書の対応セクションを、推測なしに具体的に書けるだけの情報がある
- 曖昧なポイント・未確認の前提が残っていない
- ユーザーが意図的に「対象外」「該当なし」と明言している場合のみスキップ可
- **わからないことがわからない状態は最も危険**。トピックごとに「このトピックで自分がまだ知らないことはないか？」とセルフチェックする

## 機能仕様書のヒアリング

### バッチ 1: 基本情報

- 機能名は？
- どんな課題を解決する？（1〜2文）
- 対象ユーザーは？
- スコープは？（対象範囲 / 対象外）

### バッチ 2: 機能の全体像

- この機能の主な処理フローは？（入力 → 処理 → 出力）
- どのような画面やインターフェースがある？
- どのようなデータを扱う？（入力データ、保存データ、出力データ）
- 外部システムとの連携はある？

### バッチ 3: フロントエンド深掘り（画面がある場合）

- ページ構成と画面遷移は？
- 主要なユーザー操作は？
- フォームがある場合、フィールドとバリデーションルールは？
- レスポンシブ対応やアクセシビリティの要件は？

### バッチ 4: バックエンド深掘り（APIがある場合）

- APIエンドポイント（メソッド、パス、概要）は？
- 認証・認可の仕様は？
- バッチ処理や非同期処理はある？
- 外部サービスとの連携仕様は？
- キャッシュやレートリミットの要件は？

### バッチ 5: データベース深掘り（データ永続化がある場合）

- テーブル構造（主要カラム、型、制約）は？
- テーブル間のリレーションシップは？
- データのライフサイクル（作成・更新・削除）は？
- 既存テーブルからのマイグレーションはある？

### バッチ 6: 品質（未回答の場合のみ）

- エラーハンドリングの要件は？
- パフォーマンスの要件は？
- 技術スタックの制約は？

## PRDのヒアリング

### バッチ 1: 背景

- プロダクト / 機能名は？
- どんな課題を解決する？（可能なら定量的に）
- 対象ユーザーは？
- スコープは？（対象範囲 / 対象外）

### バッチ 2: プロダクト目標

- 成功の定義は？（指標 / KPI）
- ユーザーペルソナとその主要ニーズは？
- 競合状況や制約は？

### バッチ 3: ソリューションとスコープ

- 提案するソリューションの概要は？
- フェーズ分けやMVPの範囲は？
- リスクや未解決の質問は？
- タイムラインやリリース制約は？

## ドメイン分割のガイドライン

ヒアリング完了後、以下の基準でドメインを特定する:

1. **独立して読める単位**にする（1ファイルで1つの関心が完結）
2. **1ファイルが100〜300行**に収まるようにする
3. 各ドメインが**フロントエンド/バックエンド/テーブル**のどれに属するか判断し、対応するテンプレートを使う
4. ファイル名は `{domain}-spec.md` のケバブケースにする

### テンプレート選択の基準

| ドメインの内容 | テンプレート |
|:-------------|:-----------|
| 画面、UI、ユーザー操作、フォーム、状態管理、表示ロジック | `frontend-spec.md` |
| API、ビジネスロジック、認証、バッチ処理、外部連携、キャッシュ | `backend-spec.md` |
| テーブル設計、スキーマ、制約、インデックス、マイグレーション | `table-spec.md` |
| 上記に当てはまらないもの | 最も近いテンプレートを選び、セクションを自由に追加・削除 |
