# ヒアリングパターン - システム設計書

## 共通ルール

### 質問の原則

- 1回の AskUserQuestion で1〜3問。多すぎると回答が雑になる
- **ユーザーの回答を要約して確認**してから次のトピックに進む（「〜という理解で合っていますか？」）
- **曖昧シグナルを検知**したら必ず掘り下げる（下記参照）
- トピックの順序は柔軟に。ユーザーの話の流れに合わせる
- 全バッチの質問について**不明点が完全にゼロになるまで**ドキュメント生成に進まない
- バッチ質問は「最低限聞くべきこと」。回答から新たな疑問が生まれたら追加質問する

### 曖昧シグナルと対応

| シグナル | 対応 |
|:---------|:-----|
| 「たぶん」「〜だと思う」「おそらく」 | 「根拠は何ですか？」「確定していない場合、どちらに倒しますか？」 |
| 「まだ決めてない」「未定」 | 「仮で決めるとしたらどちらですか？」「この仕様書で決めてしまいましょうか？」 |
| 「なんでもいい」「お任せ」 | 選択肢を2-3個提示し「どれが近いですか？」 |
| 回答が1文だけ | 「もう少し詳しく教えてください。具体的には〜」 |
| 代替案なしの判断 | 「他に検討した案はありますか？その案を不採用にした理由は？」 |

### 十分性の判断基準

あるトピックが「十分」と判断するための基準：
- **不明点が完全にゼロ**であること。1つでも「たぶんこうだろう」と推測で埋める箇所があるなら掘り下げが足りない
- 仕様書の対応セクションを、推測なしに具体的に書けるだけの情報がある
- 曖昧なポイント・未確認の前提が残っていない
- ユーザーが意図的に「対象外」「該当なし」と明言している場合のみスキップ可
- **わからないことがわからない状態は最も危険**。トピックごとに「このトピックで自分がまだ知らないことはないか？」とセルフチェックする

## システム設計書のヒアリング

**バッチ 1: 基本情報**
- システム名は？
- このシステムの目的は？（1〜2文）
- 目標と非目標は？
- ステークホルダーは？

**バッチ 2: アーキテクチャ**
- 主要なコンポーネントは？
- 連携する外部システムは？
- 技術スタックは？（または制約）
- データモデルは？

**バッチ 3: 非機能要件と運用**
- パフォーマンス目標は？（レイテンシ、スループット）
- 可用性目標は？（SLA）
- スケーラビリティ要件は？
- セキュリティ要件は？
- デプロイ戦略は？

## ADRのヒアリング

**バッチ 1: 背景**
- 記録すべき判断は何か？
- 背景・コンテキストは？（なぜこの判断が今必要か）
- 制約や考慮すべき力学は？

**バッチ 2: 選択肢**
- 検討した選択肢は？（最低2つ）
- 各選択肢のメリット・デメリットは？
- どの選択肢を選び、その理由は？
- この判断による影響は？

## C4モデルのヒアリング

**バッチ 1: システムコンテキスト（レベル1）**
- 設計対象のシステムは？
- ユーザー / アクターは？
- 連携する外部システムは？

**バッチ 2: コンテナ（レベル2）**
- 主要なコンテナは？（アプリ、サービス、データベース、メッセージキュー）
- 各コンテナの技術は？
- コンテナ間の通信方式は？（HTTP、gRPC、メッセージングなど）

**バッチ 3: コンポーネント（レベル3）- 必要な場合のみ**
- どのコンテナをコンポーネントに分解するか？
- そのコンテナ内の主要コンポーネントは？
- 各コンポーネントの責務とインターフェースは？

注: レベル4（コード）はC4モデル仕様書では通常ドキュメント化しない。レベル3は関連する場合のみ掘り下げる。

## セクション適応ルール

- シンプルなシステム → スケーラビリティ / インフラセクションをスキップ
- 外部連携なし → 外部APIセクションをスキップ
- ADR: 常に簡潔に。1ドキュメント1判断
- C4: 関連するレベルのみ生成（例: コンテキスト + コンテナのみ必要ならコンポーネントレベルはスキップ）
- データ永続化なし → データ移行セクションをスキップ
- 非同期処理なし → メッセージキュー / イベントセクションをスキップ
